# AIBTI 用例评审报告

> 评审人：AIBTI 用例评审员
> 评审日期：2026-04-20
> 评审对象：16 人格 × 5 case (80 条) + 5 档样本量报告 + UX 故事板 + 15 条彩蛋

---

## 📊 总体评分

- **总用例数**：80 条（16 人格 × 5 场景）
- **平均传神度**：**4.1 / 5**（整体偏高，人格指纹辨识度良好）
- **边界模糊用例**：**6 个**（主要集中在 M 派短 prompt 的 D/L 判定和 A 派抽象命题的 X/E 判定）
- **场景覆盖不足**：**4 个人格**的"创意场景"存在硬凑嫌疑（CMDE / CMLX / CMDX / AMDE）
- **需要重写**：**5 个 case**（见 BOTTOM 5）
- **文档间一致性**：**良好**（UX 10 条规则基本在 _coverage 报告中落地，彩蛋触发条件可从用例四维占比推导）

## 🎯 相邻人格边界分析

| 人格对 | 核心差异 | 区分度 | 发现的模糊用例 |
|---|---|---|---|
| **AMDE vs AMLE** | D 命令 vs L 放权 | ⭐⭐⭐⭐ 清晰 | AMDE Case 4 "报错了，你自己看，搞定" vs AMLE Case 4 "跑不起来就算了，你换个思路"——语气差距大，OK |
| **AMDX vs AMLX** | D 质问 vs L 共感 | ⭐⭐⭐⭐ 清晰 | AMDX 句句反问 / AMLX 句句气口，气质完全不同 |
| **AVDE vs CVDE** | A 业务 vs C 代码 | ⭐⭐⭐⭐⭐ 极清晰 | AVDE Case 1 PRD 全业务语言 / CVDE Case 1 贴完整 Java 类，对比教学级 |
| **AVDX vs AVLX** | D 决断追问 vs L 小作文讨论 | ⭐⭐⭐⭐ 清晰 | AVDX "这场仗应该怎么打" 强势 / AVLX 列 ABC 三选 |
| **CMDE vs CMDX** | E 命令 vs X 追问 | ⭐⭐⭐ 基本清晰，有 1 个模糊 | ⚠️ CMDE Case 3 "Cache 层换 Caffeine，替掉 Guava" 放到 CMLE 也能解释——少了"咱俩/一起"确实是 CMDE，但因为**是技术提案而非修 bug**，创意场景对 CMDE 本就偏牵强 |
| **CMDE vs CVDE** | M 短 vs V 长 | ⭐⭐⭐⭐⭐ 极清晰 | 字数差距两个数量级 |
| **CVDE vs CVDX** | E 要结果 vs X 求最优 | ⭐⭐⭐⭐ 清晰 | CVDE Case 3 嘴上"聊下"手上锁死方案 / CVDX Case 3 真的摆三条路线求取舍 |
| **CVDX vs CVLX** | D 要最优 vs L 要追问 | ⭐⭐⭐⭐ 清晰 | CVDX "给出最优方案排序" / CVLX "一层层逼问下去" |
| **CVLE vs CVLX** | E 落代码 vs X 挖原理 | ⭐⭐⭐⭐ 清晰 | CVLE "走稳比走快重要"+贴完整代码 / CVLX "为啥不用 X? 那 Y 呢?" |
| **CMLE vs CMLX** | D 搭手对等 vs L 求救低位 | ⭐⭐⭐⭐⭐ 极清晰 | "咱俩一起看看" vs "救命咋整"——措辞反差强烈 |
| **AMLE vs AMLX** | E 只要结果 vs X 要发散 | ⭐⭐⭐⭐ 清晰 | "好看就行"随便 / "我们先品一品"踩刹车 |
| **AMDX vs AVDX** | M 短 vs V 长 | ⭐⭐⭐⭐⭐ 极清晰 | AMDX "微服务是谎言，给我一个词" / AVDX 三段式战略铺垫 |
| **AVLE vs CVLE** | A 讲原理 vs C 贴代码 | ⭐⭐⭐⭐ 清晰 | AVLE "Bounded Context/DDD" / CVLE 贴完整 Controller |
| **AMDE vs CMDE** | A 甩任务 vs C 给坐标 | ⭐⭐⭐⭐⭐ 极清晰 | "修了"三字 / "OrderService.pay():88 空指针，加 Optional" |

**边界模糊用例明细（共 6 个）**：
1. **CMDE Case 3**（Cache 层换 Caffeine）—— 创意场景对 CMDE 来说本就牵强，离 CMLE 对等提案也只差一个"咱俩"
2. **AMDE Case 3**（"618 活动要个名字，给我三个，定了就用"）—— "给我三个"有轻微 X 味道，但"定了就用"把 E 托住了，边界薄但仍属 AMDE
3. **CMLX Case 3**（npm install 报错）—— "帮我看下新手刚入职"本质还是求救，但"npm ERESOLVE"坐标+求救合并起来与 CMDX 只差一个"为啥"疑问词
4. **AMLX Case 5**（"退一步 我们是在造玩具还是会长大的东西"）—— 这个 prompt 放到 AMDX 也能自圆其说，主要区别在"我们"（AMLX）vs "你"（AMDX），但长期多轮语境下 AMDX 也可能用"我们"
5. **AVLE Case 5**（ES/CQRS 三层递进）—— "咱俩一层一层来，你先判断我 push back" 很协作，但整段铺陈长到接近 AVDE 暴君，L 的味道靠"咱俩"单点撑
6. **CVDX Case 5**（14 项目架构地图）—— 阶段 1/2/3/4 串行产出节奏实际上偏 CVDE，X 靠"对比 Uber/Didi/Grab 给最优"单点撑

## ⭐ 最传神 TOP 5

### 🥇 1. AMLX Case 3 —— "先品一品"（⭐⭐⭐⭐⭐）
> "先搞一个有意思的 GitHub 主页吧，娱乐下。不追求功能，追求味道。像一首诗，或者一个谜。你先不要写代码，我们先品一品。"

**理由**：人格**灵魂场景**。"味道/诗/谜/先品一品"四个词把 AMLX 的抽象气口 + 温和协作 + 发散刹车全部拿下。相邻人格任何一个都写不出"先品一品"。

### 🥈 2. CMDE Case 2 —— 狙击式命令（⭐⭐⭐⭐⭐）
> "OrderService.pay():88 空指针，加 Optional"

**理由**：16 字把 C/M/D/E 四维钉死，**文件名:行号+动词**的识别口诀一眼成立。放到任何相邻人格都会破功（AMDE 没坐标，CVDE 会铺 50 字，CMDX 会加"为啥"）。

### 🥉 3. AMDX Case 5 —— 元对话追问（⭐⭐⭐⭐⭐）
> "等等。我们聊了这么久，你有没有发现——我们在回避真正的问题？"

**理由**："等等"+破折号在凌晨 4 点切断 14 轮前文，元对话追问是 AMDX 的招牌动作。"真正的问题"不点名，留给 AI 猜——这是典型的 M+X 组合。

### 🏅 4. AVDE Case 1 —— PRD 暴君的标本（⭐⭐⭐⭐⭐）
> 三百字 PRD 铺完，结尾一个"做"字

**理由**：A（全业务语言）+ V（三段铺陈）+ D（单字命令）+ E（要产物）四维全部拉满。"做"这一个字把暴君的气质推到极致。与 CVDE 对照简直是教学级边界样本。

### 🏅 5. CVLX Case 4 —— 六连追问的苏格拉底（⭐⭐⭐⭐⭐）
> "线程池→EventLoopGroup→CPU→TCP→网卡→Filter" 6 层连环"那 X 呢 / 那 Y 呢"

**理由**：紧迫 + 完整工具输出（arthas/GC/netstat/ethtool）+ 6 层追问 + "陪我一起挖""我照着做做完回来跟你对"。协作+探索+具象+冗长同时拉满，是 CVLX 最纯粹的形态。

## ⚠️ 需要重写 BOTTOM 5

### ❌ 1. CMDE Case 3 —— 硬凑的"创意场景"（⭐⭐）
> "Cache 层换 Caffeine，替掉 Guava"

**问题**：
- CMDE 是**祈使句机枪手**，不是创意人格。硬塞一个"创意/讨论"场景是 5 场景强制覆盖带来的副作用
- 这条本质就是**日常技术提案**，和 Case 1 "加 email getter" 气质重叠（都是 C+M+D+E 命令短句）
- 放到 CMLE 只要加一句"咱俩换一下" 就成立

**建议重写**：
- 方案 A：把"创意场景"改成"架构圆桌会议上 CMDE 的一句锋利打断"，加入时空背景——`"都别 PPT 了，prod 先把 Guava 换 Caffeine 压一周看看。"`（保留短命令但带会议场景）
- 方案 B：承认 CMDE 本就极少出现在创意场景，把 Case 3 改为**"代码评审时丢一条极简 reject"**——`"PR-423 rejected，命名不一致，重提"`（更贴合 CMDE 执行肌肉）

### ❌ 2. CMLX Case 3 —— 同样硬凑的创意（⭐⭐）
> "npm install 报 ERESOLVE peer dep 冲突 帮我看下 新手刚入职"

**问题**：
- CMLX 是**卡点求助侠**，根本不具备"创意"肌肉。这条 prompt 本质还是 Case 1 的复刻（小 bug 求助），只是加了个"新手"的身份标签
- 放到 Case 1/4/5 的上下文里也完全成立

**建议重写**：
- 改为**"新人对技术选型的迷茫式求助"**：`"学长问下，我们为啥选 Kafka 不用 Pulsar？我看文档看懵了 完全没头绪"`（保留求救低位，但从 bug 排查上升到选型讨论，算弱探索）

### ❌ 3. CMDX Case 3 —— 伪创意（⭐⭐⭐）
> "Redis QPS 每 5 分钟掉一次零，持续 3 秒，啥原因？没定时任务啊"

**问题**：
- 这条本质是**现象排查**（和 Case 1/4 同质），不是"开放讨论/创意"
- CMDX 的 X 轴全靠"啥原因"撑着，五个 case 全是"为啥/啥原因/咋回事"疑问句，五条案例气质重复度 > 60%

**建议重写**：
- 改为 CMDX **"对某个业界做法的反常识追问"**：`"大厂都用 Kafka 不用 Pulsar？Pulsar 的分层存储明明更优，为啥没普及？"`（从 bug 排查转到技术趋势追问，给探索维度真正拉开）

### ❌ 4. AMDE Case 3 —— 边界偏软（⭐⭐⭐）
> "618 活动要个名字，给我三个，定了就用。"

**问题**：
- "给我三个"轻微带 X 味（类似让 AI 出候选）——虽然"定了就用"托住了 E，但 AMDE 的核心气质是**"我只要一个结果"**，"要三个"有点松口
- 相比 Case 1/2/4/5 的纯 E，这条是整组最软的一条

**建议重写**：
- 改为**更典型的甩锅式创意**：`"618 活动名字你起一个，我发群里直接用。"`（彻底只要一个，不给选择空间，把 E 拉满）

### ❌ 5. AVLE Case 5 —— 协作意味被铺垫淹没（⭐⭐⭐）
> ES/CQRS 三层递进，"咱俩一层一层来"

**问题**：
- 三层铺陈叠加 + 每层都是完整段落，整体篇幅已经接近 AVDE 的暴君感
- L 的味道完全靠结尾"咱俩一层一层来你先判断我 push back"一句话撑，前面 90% 的文本没有协作气口
- 如果 LLM 只读前 80%，大概率会判成 AVDE

**建议重写**：
- 不动核心内容，**在每层里加入一句真正的协作气口**：`"第一层你先给判断，说错也没事——我自己也拿不准 ES 在风控里是不是过度设计"`（让协作分布在三层而不是塞尾巴）

## 🧩 场景覆盖缺口

### 硬凑场景的人格（建议弱化"创意"维度权重）

| 人格 | 创意场景硬凑度 | 建议 |
|---|---|---|
| **CMDE** | 高 | 该人格本就不适合创意。建议把 5 场景改为"日常/压力/提案/失败/多轮"，用"提案"替代"创意" |
| **CMLX** | 高 | 该人格本就是低位求救。把"创意"替代为"对技术选型的迷茫式求助" |
| **CMDX** | 中 | Case 3 和 Case 1/4 同质，建议改为追问技术趋势而非追问某个现象 |
| **AMDE** | 中 | Case 3 软了一点，改成"直接甩一个就行"会更纯 |

### 真实但未覆盖的场景

1. **午饭后的疲倦提问**（13:00-14:00 "脑子卡住求救"）—— 和"深夜"对照的另一种疲劳场景，现在 80 条里没有
2. **被反复 code review 打回来的复仇式 prompt**—— "review 第 4 次打回，这次你告诉我到底要我怎么写"——能很好区分 AMDE 和 CVDE
3. **跨时区协作场景**——"美国同事 5 小时后起床就要看到 demo"——能测"紧迫感+清晰度"的组合
4. **技术文章阅读后的 aha moment 提问**——"刚看完 DeepSeek-V3 paper，MoE 专家数真的越多越好吗?"——适合 CVLX/AVDX/AMLX 发挥

## 🔄 重复/冗余清单

### 同人格内 case 重复

| 人格 | 重复点 |
|---|---|
| **CMDX** | Case 1/2/3/4/5 全部以"为啥/咋回事/啥原因"结尾，5 条 case 的"探究开口"模式重复度 > 80%，建议至少 2 条改成**带假设的证据式追问**（如 "我怀疑是 xxx，你验证下"） |
| **CMDE** | Case 1/4/5 都是"文件名:行号+动词"的代码操作，Case 2（空指针）和 Case 5（addFilterBefore）气质差不多 |
| **AMLE** | Case 2（"我看不动了你来吧"）和 Case 4（"跑不起来就算了"）都是"疲软放权+换思路"，只差场景包装 |
| **CVDE** | Case 1/2/3/5 都是"贴完整代码/规格+硬命令"四段式，只是场景换了皮（重构/上线/积分/迭代），结构同质化严重 |

### 跨人格 prompt 重复

- **"紧急 P0 线上 bug"场景**被 AVDE Case 2 / AVLX Case 2 / CVDE Case 2 / CMLX Case 2 / CVDX Case 2 / CVLX Case 2 / CMLE Case 2 / CMDX Case 2 / CMDE Case 2 都用过，共 9 个人格的"压力场景"都落在**生产故障**上。建议拆开：一半改成**"个人 deadline 压力"**（如"明早答辩"/"老板下午要看 demo"），丰富压力多样性
- **"Kafka/Redis 相关 bug"**出现在 7+ 个 case 里，建议引入**前端/数据工程/ML/运维/Go/Rust 等其他技术栈**的报错场景

## 📋 综合建议

### 给 Skill 判定逻辑的改进点

1. **相邻人格分级权重**：LLM 在判定时，**除了打分还要做相邻人格对照**。例如判为 CMDE 时，同时给出 "vs CMLE / CMDX / CVDE / AMDE 各差多少分"，便于 DEBUG 和边界 case 复盘
2. **D/L 轴的判定信号需扩展**：目前 D/L 判定重度依赖"咱俩/一起/咋办"等关键词。建议增加**句式级信号**（祈使句 vs 疑问句 vs 陈述句占比）和**主语人称信号**（第一人称"我" vs 第二人称"你" vs 第一人称复数"我们"占比）
3. **短 prompt 的 D/L 判定容易失真**：M 派（< 60 字）的用例里，L 的判定几乎全靠单个关键词（"求""搭把手""帮我"），建议 M 派单独校准阈值
4. **场景权重需适配人格**：CMDE/CMLX/CMDE 这些"执行/求救"型人格在"创意场景"本就稀薄，不应强求 5 场景均衡——可以把"创意"替换为"技术提案"或"反常识追问"

### 给 UX 改造的补充建议

1. **AMLX 的彩蛋 T2-1 "赛博诗社社长" 判定条件过宽**（AMLX ≥ 70%）—— 实际 AMLX case 都是 < 50 字的短句气口，加上 70% 这个门槛，可能普通碎念型用户也会触发。建议再加一条"创造类+战略类占比 ≥ 30%"的副条件
2. **反甩锅者 T4-3 彩蛋必须保留**——从 AMDE Case 文案看（"甩锅侠""锅越甩越大"），主人格页如果按 UX 故事板改成"高效委派型"，那 AMDE 低占比胜出的用户很容易感到冒犯。T4-3 是必要的安全网
3. **UX 故事板 10 条改进在 _coverage 报告中落地率**：
   - ✅ 正向命名（_coverage 档位 3 用的是"祈使句机枪手"但配 🔫 图标 + "精准定位，精准开火"slogan，语气正向，OK）
   - ✅ 样本不足不出结论（档位 1/2 严格执行）
   - ✅ 温柔措辞（"还不到下判断的时候"而非"失败"）
   - ⚠️ **超能力:暗面 2:1 比例**在 _coverage 的"诊断"部分没有显式体现——当前诊断是"依据 Karpathy"这种中性的，没有明确的"超能力段+暗面段"结构。建议档位 4/5 的主人格块加一个 `✨超能力 / ⚠️暗面 / 📖一句话` 三段结构
   - ⚠️ **大神理论出处**在档位 3/4/5 中有体现但引用格式不统一（有的是"依据: Karpathy 'LLM is collaborator'"，有的是"Wei 2022《CoT》"），建议统一成 `[作者][年份]《标题》` 格式

### 开源前必须修的 P0 问题

- **P0-1**：CMDE Case 3 / CMLX Case 3 的创意场景硬凑问题——新用户读到会觉得"这 5 条案例怎么有一条怪怪的"，影响对人格描述的信任度
- **P0-2**：CMDX 5 条 case 全以"为啥"结尾的重复度问题——开源后社区第一个会挑这个
- **P0-3**：CVDE 5 条 case 同质化严重（都是贴代码+命令四段式）——CVDE 应至少有一条"需求方反悔式命令"（上一轮让改 A，这轮又要改回 B）拉开差异
- **P0-4**：AMDE Case 3 软边界问题——开源前必须改成"只要一个"的纯 E 形态，否则容易被当反例
- **P0-5**：AMLX 的彩蛋触发条件需要收紧，避免普通碎念型用户被错判为"赛博诗社社长"
- **P1-6**：跨人格"Kafka/Redis 故障"场景过度集中，开源后有"样本单一"的批评风险——建议上线前至少替换 3 条为其他技术栈

## ✅ 整体结论

**用例库整体达到可上线标准（传神度 4.1/5），边界模糊和重复问题集中在 6 个 case（7.5%），均可通过轻量重写修复，不伤及人格指纹。**

**修完 P0-1 到 P0-5 这 5 条后可直接开源。** 人格描述的清晰度、相邻对的区分度、UX 的温度都明显好于同类 MBTI 工具，"先品一品"（AMLX）、"加 Optional"（CMDE）、"我们是否在回避真正的问题"（AMDX）这三句都具备成为 meme 的传播潜力——这是这份用例库**最大的护城河**。
