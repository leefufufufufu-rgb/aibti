# AVLE · 架构布道士 - 5 Cases

> 代码：AVLE（Abstract + Verbose + Collaborative + Execute）
> 中文名：架构布道士 ⛪
> 英文名：DDD Evangelist
> Slogan：「讲完概念，一起干」
> 特征：先铺一层理论（DDD / 六边形 / Clean Architecture / CQRS），再拉 AI 一起落地——协作，但必须出东西。

---

### Case 1: 日常工作 —— 订单模块落地 DDD
**场景**: 日常迭代中想把一个订单服务按 DDD 重构，先讲一遍理论再拉 AI 共同推进。
**Prompt**:
```
来，我们今天把订单模块按 DDD 认真做一遍。先对齐一下心智模型：领域驱动设计的核心不是分层，而是围绕"限界上下文（Bounded Context）"切分业务语言，让代码和业务用同一套 Ubiquitous Language 说话。我们订单这边至少能识别出三个上下文——交易上下文、履约上下文、结算上下文，它们之间通过领域事件（Domain Event）解耦，而不是互相直接调 service。

在战术层面，Order 是聚合根，OrderItem 是内部实体，金额、折扣这类是值对象（Value Object），绝对不能逸出。仓储（Repository）只暴露聚合根，应用服务（Application Service）只做用例编排，不写业务规则。基础设施层用六边形架构那套 Port & Adapter 把 DB、MQ 都挡在外面。

我已经把现有 OrderService.java 的代码贴在下面了。咱俩一起搞一下：你先按这个模型帮我画出聚合边界和领域事件，然后我们一块儿把 createOrder 这个方法重构成应用服务 + 聚合根的形态，我来 review 边界是否纯净。走起。
```
**四维判定**:
- A/C: **A** —— 通篇"限界上下文/聚合根/值对象"，抽象术语堆叠。
- M/V: **V** —— 明显铺陈，先讲心智模型再进具体任务。
- D/L: **L** —— "咱俩一起搞一下""你先…然后我们一块儿"，明显协作。
- X/E: **E** —— 要求真改 `createOrder`，产出代码。
- **最终代码**: **AVLE**

**为什么典型**: AVDE 会说"按 DDD 重构，直接给我代码，不要解释"；CVLE 会直接贴 code "把这个 if 拆成 strategy"；本 case 先布道再邀请共建，是布道士的典型姿态。

### Case 2: 压力/紧迫 —— 用理论救一个快崩的项目
**场景**: 项目线上事故频发，负责人用 Clean Architecture 给团队（含 AI）打气并要求立即动手整改。
**Prompt**:
```
不瞒你说，这个项目现在已经快失控了：P0 告警一天三次，改一个按钮要动五个服务，测试覆盖 12%。但我不想用"加班加人"这种反模式来救——那只会让腐坏加速。我们得用 Clean Architecture 的思路做一次"外科手术式"的抢救。

Uncle Bob 讲过，软件腐烂的根因是依赖方向错了：业务规则依赖了框架、ORM、HTTP。我们现在就是典型的"Entity 依赖 Controller"的反向结构。当务之急是把 Use Case 层立起来，让依赖只能从外（frameworks）指向内（entities），中间用 Boundary 接口倒置。DIP（依赖倒置）是这一切的支点。

我知道你看到这个烂摊子会想"先止血再说"，但我坚持先有模型再动刀，否则是用混乱救混乱。这样——我们今晚一起干：我负责把当前调用链画给你看并指出哪条边违反了依赖规则，你负责把最痛的那条（订单→支付→通知）按 Use Case + Boundary 重写成一份可以直接合入的 PR。上，别怂。
```
**四维判定**:
- A/C: **A** —— "依赖规则/Boundary/DIP"全是抽象原则。
- M/V: **V** —— 紧迫之下依然长篇铺理论而不是一句"快改"。
- D/L: **L** —— "我们今晚一起干""你负责…我负责…"，分工协作。
- X/E: **E** —— 目标是一份可合入的 PR，必须出代码。
- **最终代码**: **AVLE**

**为什么典型**: AVDE 压力下会变成"现在，立刻，按 Clean 写，给我 PR"；AVLX 会写一堆方案 A/B/C 让别人选；布道士哪怕着火也要先讲完依赖方向，然后拉 AI 并肩冲。

### Case 3: 开放讨论/创意 —— 理论与实践碰撞
**场景**: 在做一个新系统前期，想用六边形架构 + 事件风暴开一次"思想碰撞"，然后动手搭骨架。
**Prompt**:
```
我想跟你做一次真正的 architectural conversation，而不是 code-first 直接开撸。我们在设计一个 AI 陪练系统——多模态输入、流式输出、记忆层、评估层。摆在我面前的选择空间很大，我更想先确立 mental model。

我个人倾向六边形架构（Ports & Adapters）：核心是"领域"——对话会话、评估策略、成长画像；外圈是 adapter：LLM provider、TTS、前端 WS、向量库。这样任何一个 LLM 换供应商、任何一个前端换形态，都不影响核心。同时我想用事件风暴（Event Storming）来识别领域事件：SessionStarted、UtteranceEvaluated、CoachingStrategyChanged……这些事件会成为 CQRS 里 write model 和 read model 的分界。

来，咱俩互相挑战一下。你先说：这种抽象在 AI 系统里有没有 over-engineering 的风险？哪些边界是"假边界"？然后我们对齐完之后，一起动手产出一份骨架目录 + 核心接口定义（至少 Port、Adapter、Domain Event 三类），今天就要能跑起 hello world。
```
**四维判定**:
- A/C: **A** —— Ports & Adapters、事件风暴、CQRS，全抽象。
- M/V: **V** —— 铺大段背景与倾向。
- D/L: **L** —— "咱俩互相挑战""我们对齐完之后一起动手"。
- X/E: **E** —— 要目录骨架 + 接口定义 + hello world，必须有产物。
- **最终代码**: **AVLE**

**为什么典型**: AVDX 讨论完可能就停在"选哪种架构更好"；CVLE 上来就写 `class SessionService`；布道士讨论是工具，落地才是目的——这正是 E 轴的标志。

### Case 4: 失败/卡壳 —— "这不符合 XX 原则，我们怎么改"
**场景**: AI 按需求给了一版代码，作者用 SOLID 诊断出违规，邀请一起改。
**Prompt**:
```
等一下，这版实现我看了几遍，越看越不舒服——不是 bug，是"气味"。我们停下来用 SOLID 过一遍，它明显违反了至少两条原则。

第一，单一职责（SRP）：你把 OrderCreator 这个类既做参数校验、又做库存扣减、又做通知、还写审计日志。按 Uncle Bob 的定义，类应该"只对一个 actor 负责"——这四件事背后是四个不同的业务干系人（运营、仓储、营销、合规），所以它至少要拆成四个合作者。第二，开闭原则（OCP）：notify() 里写死了 if type == 'sms' else 'email' else 'wecom'，以后再加一种渠道就得改这个类，这是典型的"对扩展封闭"反向实现。

我不想直接让你"改一下"——那样下次还会犯。我们一起返工：第一步，你跟我一起重新用 SRP 梳理 OrderCreator 的职责清单（list 出来）；第二步，我们把通知部分按 OCP 改成策略模式 + 注册表；第三步，你把改好的代码贴出来，我负责拿原则逐条打勾。来吧，这次咱俩把它做对。
```
**四维判定**:
- A/C: **A** —— SRP、OCP、"对扩展封闭"，全抽象原则。
- M/V: **V** —— 不直接丢结论，铺了完整诊断。
- D/L: **L** —— "我们一起返工""咱俩把它做对"，不是命令而是协作返工。
- X/E: **E** —— 终点仍是改好的代码，用原则打勾收尾。
- **最终代码**: **AVLE**

**为什么典型**: AVDE 会说"违反 SRP，重写，不许问"；AMLE 会简洁到"这里拆一下"；布道士一定要先把"为什么不符合"讲透，再拉你一起修——这是"布道 + 协作 + 要结果"三位一体。

### Case 5: 复杂多轮对话 —— 层层深入的理论应用
**场景**: 围绕一个核心子域，多轮深挖 DDD + CQRS + Event Sourcing 的组合落地。
**Prompt**:
```
我们上一轮已经对齐了"风控"是这个系统的 Core Domain，不是 Supporting 也不是 Generic。基于这个判断，我想跟你再深入三层，最后收口到代码。

第一层（战略）：既然是 Core Domain，它就值得用最重的战术武器。我倾向于 Event Sourcing——风控场景天然需要可追溯、可回放、可审计，状态导向的模型（CRUD）会把故事讲丢。你怎么看？有没有"明知是 Core 也不该上 ES"的反例？

第二层（模型）：如果上 ES，那写模型就是一连串 RiskEvaluated / RuleHit / DecisionMade 事件，读模型要用 CQRS 单独投影——比如"近 7 天命中规则排行榜"就是一个独立 projection，跟写侧解耦。我想让你跟我一起把事件清单先定下来（至少 8 个），并标出哪些是 domain event、哪些是 integration event。

第三层（落地）：事件清单定了之后，我们一块儿出代码——聚合根 RiskCase 用 EventSourcedAggregate 基类，Repository 只跟 event store 打交道，projection 用一个独立的 handler 模块消费。今天的目标是跑通"一次评估事件 → 两个 projection 更新"的最小闭环。

咱俩一层一层来，每层你先给判断+理由，我 push back，对齐了再往下走。最后一定要出可运行代码，不要停在纸上。
```
**四维判定**:
- A/C: **A** —— Core/Supporting 子域、ES、CQRS、projection，重度抽象。
- M/V: **V** —— 三层递进、每层铺陈，典型长篇。
- D/L: **L** —— "咱俩一层一层来""你先判断，我 push back"，对等协作。
- X/E: **E** —— 明确"最后一定要出可运行代码，不要停在纸上"。
- **最终代码**: **AVLE**

**为什么典型**: AVLX 会把三层都变成"方案 A / 方案 B 请选择"；CVDE 会直接开始写 `class RiskCase extends EventSourcedAggregate`；布道士是"每一层都讲透、每一层都一起走、终点必须落代码"——这正是架构布道士区别于 PM 小作文和技术导师的本质。
